# Электронная Цифровая Подпись

Учебный проект по предмету **"Архитектура, программное обеспечение и безопасность автоматизированных систем"**.

Демонстрирует практическое применение ЭЦП в клиент-серверном взаимодействии на двух сценариях.

---

## Сценарии

### Сценарий 1 — Подпись на стороне клиента
Клиент подписывает сообщение своим приватным ключом и отправляет на сервер для верификации.

```
Клиент                                Сервер
──────                                ──────
Генерация ключей RSA-2048
Подпись сообщения (SHA-256)
POST /verify {message, signature, public_key}  →
                                      Верификация подписи
                              ←       { verified: true/false }
Отображение результата
```

### Сценарий 2 — Подпись на стороне сервера
Сервер генерирует и подписывает сообщение, клиент верифицирует локально.

```
Клиент                                Сервер
──────                                ──────
GET /public-key                →
                               ←      Публичный ключ сервера
GET /sign-message              →
                               ←      { message, signature }
Верификация локально (без запроса)
Отображение результата
```

## Структура проекта

```
project/
├── server.py          # Flask-сервер, криптография на Python
├── client.html        # Веб-клиент с визуализацией сценариев
├── Dockerfile         # Образ для сервера
└── docker-compose.yml # Оркестрация двух контейнеров
```

### `server.py`
Flask-приложение на Python. Отвечает за:
- Генерацию ключевой пары RSA-2048 при запуске (ключи сервера живут в памяти)
- `POST /verify` — принимает сообщение, подпись и публичный ключ клиента, верифицирует подпись
- `GET /public-key` — возвращает публичный ключ сервера в формате PEM
- `GET /sign-message` — генерирует случайное сообщение, подписывает приватным ключом сервера, возвращает сообщение и подпись

Ключевые зависимости: `flask`, `cryptography`

### `client.html`
Одностраничное веб-приложение. Отвечает за:
- Визуализацию пошагового выполнения каждого сценария
- Генерацию ключей RSA и подпись сообщений прямо в браузере (JS-библиотека `forge`)
- Отправку HTTP-запросов к серверу через `fetch()`
- Верификацию подписи сервера локально, без обращения к backend (Сценарий 2)
- Демонстрацию атаки подмены — тумблеры «подделать сообщение» и «подделать подпись»

Зависимости: [`node-forge 1.3.1`](https://cdnjs.cloudflare.com/ajax/libs/forge/1.3.1/forge.min.js) через CDN

### `Dockerfile`
Описывает образ для Flask-сервера на базе `python:3.12-slim`. Устанавливает зависимости и запускает `server.py`.

### `docker-compose.yml`
Поднимает два контейнера:
- **server** — Flask на порту `5000`
- **client** — Nginx на порту `8080`, раздаёт `client.html` как статику

## Запуск

**Требования:** Docker Desktop

```bash
docker compose up --build
```

После запуска:
- Клиент: [http://localhost:8080](http://localhost:8080)
- Сервер API: [http://localhost:5000](http://localhost:5000)

```bash
# Остановить
docker compose down
```

## Алгоритм и параметры

- **Алгоритм подписи:** RSA
- **Длина ключа:** 2048 бит
- **Хэш-функция:** SHA-256
- **Схема дополнения (padding):** PKCS1v15
- **Формат ключей:** PEM
- **Передача подписи:** Base64

## Упрощения (учебный проект)

В реальных системах дополнительно применяются:

- **Сертификаты X.509** — публичный ключ подписывается центром сертификации (CA), иначе клиент не может проверить, кому он принадлежит
- **Хранение ключей клиента** — в реальности приватный ключ хранится в защищённом хранилище (токен, смарт-карта, TPM), а не генерируется заново на каждый запрос
- **Временные метки** — без них возможна replay-атака: перехваченная подпись может быть повторно использована
- **HTTPS** — весь обмен должен идти поверх TLS, иначе возможна атака man-in-the-middle с подменой публичного ключа
- **Регистрация ключей** — в Сценарии 1 сервер принимает публичный ключ прямо в запросе; в реальности он должен быть зарегистрирован заранее
